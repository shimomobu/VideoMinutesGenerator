# requirements.md

# 動画議事録生成システム 要件定義書 v0.1

## 1. 文書目的

本書は、新規プロジェクト「動画議事録生成システム」の初期成果物として、システムの目的、対象範囲、要求仕様、MVP範囲、設計方針を定義するものである。  
本書を起点として、以後のアーキテクチャ設計、タスク分解、実装、テスト設計を行う。

---

## 2. システム概要

### 2.1 システム名
動画議事録生成システム

### 2.2 目的
会議動画から音声を抽出し、文字起こし・要約・決定事項抽出・ToDo抽出を行い、標準化された議事録を自動生成する。

### 2.3 想定利用者
- 会議参加者
- PM / PMO
- 管理職
- 事務局
- プロジェクト担当者

### 2.4 解決したい課題
- 議事録作成に手間がかかる
- 作成者によって品質にばらつきがある
- 決定事項やToDoの抜け漏れが発生する
- 会議内容の再確認に時間がかかる

### 2.5 提供価値
- 議事録作成時間の削減
- 記録品質の標準化
- 決定事項・保留事項・ToDoの構造化
- 動画と議事録の追跡性向上

---

## 3. 対象範囲

### 3.1 対象
- 会議動画ファイルを入力として受け付ける
- 音声抽出を行う
- 文字起こしを行う
- 発言内容を解析し、議事録を生成する
- Markdown および JSON 形式で出力する

### 3.2 対象外
- リアルタイム字幕生成
- 完全自動の本人ベース話者特定
- 法的証跡としての保証
- 外部システムとの高度なワークフロー連携
- 初期リリース時点での DOCX / PDF 自動出力

## 6. 非機能要件

| 項目 | 要件 |
|---|---|
| 性能 | 60分程度の動画を実用的な時間で処理できること |
| 再実行性 | 途中失敗時に再実行可能であること |
| 保守性 | 音声認識・要約・出力フォーマッタを差し替えやすいこと |
| 拡張性 | 出力形式や抽出ロジックを追加しやすいこと |
| セキュリティ | 機密動画を外部送信しない構成を選択可能であること |
| 監査性 | 元データと生成結果の対応関係を保持できること |
| 可観測性 | 各処理の実行ログ、失敗箇所、処理時間を確認できること |

---

## 7. 処理フロー

1. ユーザーが会議動画を投入する  
2. 動画から音声を抽出する  
3. 必要に応じて音声前処理を行う  
4. 音声認識で文字起こしを行う  
5. 話者単位または発話単位に整形する  
6. テキスト解析により要約・議題・決定事項・保留事項・ToDoを抽出する  
7. 議事録テンプレートに整形する  
8. Markdown / JSON で保存する

## 8. 議事録フォーマット要件

### 8.1 初期採用フォーマット
初期リリースでは「標準議事録形式」を採用する。

### 8.2 標準議事録形式

| セクション | 内容 |
|---|---|
| 1. 会議情報 | 会議名、日時、参加者、元動画情報 |
| 2. 会議要約 | 3〜5行程度の全体要約 |
| 3. 議題 | 主な議題一覧 |
| 4. 議論内容 | 議題ごとの要点整理 |
| 5. 決定事項 | 合意済み事項 |
| 6. 保留事項 | 継続検討事項、未決事項 |
| 7. ToDo | タスク、担当、期限、備考 |
| 8. 参考ログ | 必要に応じた発話抜粋 |

## 9. 内部データ構造要件

```json
{
  "meeting_info": {
    "title": "",
    "datetime": "",
    "participants": [],
    "source_file": ""
  },
  "summary": "",
  "agenda": [],
  "topics": [
    {
      "title": "",
      "points": []
    }
  ],
  "decisions": [],
  "pending_items": [],
  "todos": [
    {
      "task": "",
      "owner": "",
      "due_date": "",
      "notes": ""
    }
  ],
  "transcript": [
    {
      "start_time": "",
      "end_time": "",
      "speaker": "",
      "text": ""
    }
  ]
}
```

要件:
- 元動画情報と出力結果の対応が取れること
- transcript を基点に各抽出結果を再生成できること
- 出力フォーマッタは transcript / summary / decisions / todos を利用して独立動作できること

## 11. 品質上の前提・制約

| 項目 | 内容 |
|---|---|
| 話者識別 | 初期は Speaker A/B 等の擬似識別でも可 |
| 固有名詞認識 | 人名、製品名、社内用語は誤認識しやすい |
| 決定事項抽出 | 曖昧表現からの誤抽出リスクがある |
| 期限抽出 | 「来週」「月末」などの曖昧表現に注意が必要 |
| 最終責任 | 議事録の最終確定は人手レビュー前提とする |

## 12. 開発方針

| 項目 | 方針 |
|---|---|
| 開発方式 | MVP先行 |
| 優先順位 | 完全自動高精度より、まず使えるレベルの自動議事録生成を優先 |
| 設計方針 | 前処理、ASR、解析、フォーマッタを疎結合にする |
| 安全方針 | 元データ・中間データ・出力データを分離保存する |
| テスト方針 | 各段階で単体テスト可能な構成にする |

## 14. テスト観点

| 区分 | 観点 |
|---|---|
| 入力 | 対応形式以外の動画を拒否できるか |
| 音声抽出 | 音声なし動画、長時間動画を扱えるか |
| 文字起こし | タイムスタンプが保持されるか |
| 話者分離 | 単一話者・複数話者で破綻しないか |
| 要約 | 空文字や短文会議で異常終了しないか |
| 決定事項抽出 | 決定事項なし会議を正しく扱えるか |
| ToDo抽出 | 担当不明・期限不明でも扱えるか |
| 出力 | Markdown / JSON の構造が壊れないか |

## 15. リスクと対策

| リスク | 内容 | 対策 |
|---|---|---|
| R-01 | 音声認識誤り | 固有名詞辞書、手動修正、再生成を可能にする |
| R-02 | 話者分離精度不足 | 初期は擬似話者許容にする |
| R-03 | 抽出誤判定 | 決定事項・ToDoの抽出根拠を transcript に紐付ける |
| R-04 | 長時間動画処理負荷 | 分割処理、再実行機構を設ける |
| R-05 | 出力形式追加時の改修増大 | フォーマッタを独立モジュール化する |
