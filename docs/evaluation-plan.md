# evaluation-plan.md

# 動画議事録生成システム 実データ評価計画 v0.1

## 1. 目的と位置付け

本書は MVP 実装（TASK-00-01〜TASK-07-04）完了後に行う実データ評価の計画を定める。

目的は以下の2点である。

1. 実際の会議動画を入力として、パイプライン全体が期待どおり動作するかを確認する
2. 生成される議事録の品質を定量・定性的に評価し、改善の優先度を決定する

本書は `requirements.md` §8（出力仕様）および `test-plan.md` の観点を実データ評価に展開したものである。

---

## 2. 実データサンプルの種類と収集方針

### 2.1 サンプル分類

| 分類 | 説明 |
|---|---|
| **標準ケース** | 音声品質が良好、参加者 2〜5 名、10〜30 分程度の会議 |
| **大規模ケース** | 参加者 6 名以上、または 30 分超の長尺会議 |
| **短尺ケース** | 5 分未満、または議題が 1〜2 件程度の短い録音 |
| **低品質ケース** | 雑音・エコー・音量ムラなど、音声品質が低い録音 |
| **境界ケース** | 無音、ノイズのみ、会話なしなど、ASR が期待どおりに動かない可能性があるケース |

### 2.2 収集方針

- 実データには個人情報（氏名・社名・プロジェクト名など）が含まれる場合がある。評価に使用する前に、内容の確認と必要に応じた匿名化処理を行うこと。
- 動画ファイルは `tests/fixtures/real/` には置かず、別途指定した評価用ディレクトリに保管する（§7 参照）。
- 評価対象とするファイルは `eval/samples/` 以下の `README.md` にリスト管理する。

## 3. 評価観点

各観点は 4 段階で採点する。

| スコア | 意味 |
|---|---|
| 4 | 期待を超える品質 |
| 3 | 期待どおり（実用水準） |
| 2 | 概ね動作するが要改善 |
| 1 | 動作するが品質が低い |
| 0 | 出力なし、またはクラッシュ |

---

### 3.1 要約品質

- 会議の目的・結論が 1〜3 文で簡潔にまとめられているか
- 重要な発言が漏れていないか
- 無関係な発言が混入していないか
- 日本語として自然な文章か

---

### 3.2 議題抽出精度

- 実際に議論された議題がすべて列挙されているか
- 議題の順番が実際の会議と一致しているか
- 議題として挙がっていない話題が誤って含まれていないか

---

### 3.3 決定事項抽出精度

- 会議中に合意・決定した事項がすべて含まれているか
- 「決まっていないこと」が誤って決定事項に分類されていないか
- 重複した記載がないか

---

### 3.4 保留事項抽出精度

- 明示的に「持ち越し」「次回確認」とされた事項が含まれているか
- 決定事項と保留事項が正しく分類されているか

---

### 3.5 ToDo 抽出精度

- 担当候補（`owner_candidate`）が発言内容から適切に推定されているか
- 期限候補（`due_date_candidate`）が「来週」「月末」等の曖昧表現も含めて保持されているか
- 確定していない担当・期限が「候補」として扱われているか（確定値と混同していないか）
- 漏れ・誤抽出の件数

---

### 3.6 Markdown 品質

- `# 議事録` から始まり、全 8 セクションが存在するか
- 各セクションのヘッダ・テーブル・リストが正しい Markdown 文法か
- レンダリングして読みやすいか
- タイムスタンプ（`[HH:MM:SS]` 形式）が参考ログに含まれているか

---

### 3.7 JSON 品質

- `MinutesOutput` スキーマに準拠しているか（`model_validate_json` でエラーなし）
- 全フィールドに期待する型・値が入っているか
- `minutes.json` → Markdown 再生成が可能なだけの情報が含まれているか
- `manifest.json` に `job_id`・`generated_at`（ISO8601）・`files` が正しく記録されているか

---

### 3.8 実行性能・運用性

#### 処理時間

- 音声抽出（FFmpeg）にかかった実時間（動画長比）
- 文字起こし（Whisper）にかかった実時間（音声長比）
- Ollama 呼び出しにかかった実時間（チャンク数含む）
- パイプライン全体の所要時間

目安として以下を記録し、ボトルネックを特定する。

```
動画長     : X 分
preprocess : Y 秒
asr        : Z 秒
analysis   : W 秒（API 呼び出し N 回）
合計       : T 秒
```

#### 再実行性

- 中間ファイル（`audio.wav`・`transcript.json`・`analysis.json`）が正しく保存されているか
- 同一 `job_id` での再実行時に、ステージスキップが正常に機能するか
- `--force` 指定時に全ステージが再実行されるか

#### エラー時の復旧しやすさ

- 失敗ステージが `logs/{job_id}.jsonl` から特定できるか
- 中間ファイルを残した状態で再実行した場合に、失敗ステージのみ再実行されるか
- `PipelineError` のメッセージがトラブルシュートに十分な情報を含んでいるか

## 4. 手動評価の手順と採点基準

### 4.1 手順

1. 評価対象の動画を評価用ディレクトリに配置し、`eval/samples/{case_id}/meta.yaml` にメタ情報を記録する
2. パイプラインを実行し、`data/output/{job_id}/` の出力を取得する
3. 動画を視聴し、議事録の内容と実際の会議を照合する
4. 各評価観点（§3.1〜3.8）を 0〜4 で採点し、コメントを記録する
5. 評価結果を `eval/results/{case_id}/eval.yaml` に保存する

### 4.2 評価結果フォーマット（eval.yaml）

```yaml
case_id: CASE-01
job_id: job_20260423_abc123          # 実行時に発行された job_id
evaluated_at: 2026-04-23T10:00:00+09:00   # ISO8601
evaluator: reviewer_name
scores:
  summary: 3          # 0〜4 の整数
  agenda: 3
  decisions: 2
  pending_items: 3
  todos: 2
  markdown: 4
  json: 4
  performance:
    duration_seconds:
      preprocess: 5
      asr: 42
      analysis: 18
      total: 67
    skip_works: true   # ステージスキップが機能したか
    error_recovery: 3  # エラー時の復旧しやすさ（0〜4）
comments:
  summary: "会議の目的は捉えているが結論が弱い"
  todos: "担当候補の推定精度が低い（発言が曖昧なケースが多い）"
  other: ""
```

## 5. 将来の半自動評価に向けたメモ

現時点では手動評価のみとするが、以下の方向で半自動化を検討する。

### 5.1 参照回答（Reference Answer）ベースの自動評価

- 評価者が「正解の議事録」をあらかじめ作成し、生成結果と差分をスコアリングする
- 決定事項・ToDo は集合比較（Precision / Recall）で数値化できる
- 要約は ROUGE スコアや LLM-as-a-judge で評価可能

### 5.2 LLM-as-a-judge

- LLM-as-a-judge ツール（Ollama 等）に評価用プロンプトを送り、生成された議事録を採点させる
- 評価コストがかかるため、定期評価（週次・リリース前）に限定する
- 評価プロンプト自体もバージョン管理する

### 5.3 回帰テストへの組み込み

- スコアが閾値（例: 全観点平均 2.5 以上）を下回った場合に CI で警告を出す仕組みを設計する
- ただし API コストの観点から、slow テストと同様に手動実行を基本とする

## 7. 評価ログの保存方針

### 7.1 ディレクトリ構成

```
eval/
  samples/
    README.md          # サンプル一覧（ファイルパスは記載しない）
    CASE-01/
      meta.yaml        # サンプルの概要（参加者数・時間・音声品質等）
    CASE-02/
      meta.yaml
    ...
  results/
    CASE-01/
      eval.yaml        # 評価結果（§4.2 フォーマット）
    CASE-02/
      eval.yaml
    ...
```

### 7.2 実データ動画の保管分離

**実データ動画ファイルはリポジトリに含めない。**

- 動画ファイルは個人情報・機密情報を含む可能性があるため、評価結果（`eval/results/`）とは保管場所を分離する
- 動画ファイルは評価担当者のローカル環境または社内限定ストレージに保管し、`eval/samples/{case_id}/meta.yaml` にはファイル名・長さ・取得日などのメタ情報のみ記録する
- リポジトリの `.gitignore` に `eval/samples/**/*.mp4`・`eval/samples/**/*.mov`・`eval/samples/**/*.mkv` を追加して誤コミットを防ぐこと

### 7.3 評価結果の管理

- `eval/results/` はリポジトリで管理し、品質推移を追跡可能にする
- 個人名・社名などが `eval.yaml` の `comments` に含まれないよう注意する
- 評価結果に問題のある情報が含まれた場合は、コミット前にマスキングする
